## Архитектура WebSocket

### Принципы работы

PowerApp использует Pusher WebSocket для real-time коммуникации между API и мобильными приложениями:

```
┌─────────────────┐    WebSocket  ┌──────────────────┐    HTTP/MQTT    ┌─────────────────┐
│   Mobile App    │◄─────────────►│   API Server     │ │◄─────────────►│   Middleware    │                  │
│   (Flutter)     │   Real-time   │   (Laravel)      │   Station      │   (Station)     │
└─────────────────┘   Events      └──────────────────┘   Control      └─────────────────┘
```

### Технический стек

**Backend (Laravel API)**:
- `laravel/reverb` - WebSocket сервер Laravel
- `pusher/pusher-php-server` - Pusher PHP SDK
- `laravel/broadcasting` - Laravel broadcasting система
- `redis` - Backend для очереди событий

**Frontend (Flutter)**:
- `dart_pusher_channels` - Pusher Dart клиент
- `flutter_hooks` - Hooks для управления состоянием
- `riverpod` - State management интеграция

## Конфигурация

### Backend конфигурация

**WebSocket Server** (`/api/config/reverb.php`):

**Что делает секция `servers`:**
- Настраивает параметры запуска Reverb сервера
- `host` и `port` определяют где запускается сервер (default: 0.0.0.0:8080)
- `hostname` используется для публичного доступа (из REVERB_HOST)
- `max_request_size` ограничивает размер WebSocket сообщений (default: 10000 байт)

**Что делает секция `apps`:**
- Конфигурирует приложения, которые могут подключаться к WebSocket серверу
- Каждое приложение идентифицируется через `key`, `secret` и `app_id`
- `allowed_origins: ['*']` разрешает подключения с любых доменов
- `ping_interval: 60` и `activity_timeout: 30` управляют поддержкой соединения

**Масштабирование:**
- Секция `scaling` настраивает работу через Redis для нескольких инстансов Reverb
- По умолчанию отключено (`REVERB_SCALING_ENABLED=false`)

**Broadcasting** (`/api/config/broadcasting.php`):

**Назначение Broadcasting системы:**
- Преобразует Laravel события в WebSocket сообщения
- Авторизует private/presence каналы через подписанные запросы
- Маршрутизирует события к правильным получателям
- Обеспечивает безопасность передачи данных

**Что настраивает конфигурация `pusher` драйвера:**
- **Протокол подключения:** с помощью пакета `laravel/reverb` создается WebSocket сервер, совместимый с Pusher протоколом
- **Аутентификация:** публичный ключ для клиентов и секретный ключ для подписи авторизации
- **Идентификация приложения:** ID настраивается в `reverb.php`
- **Сетевые настройки:** адрес определяется переменными `REVERB_HOST` и `REVERB_PORT`
- **Безопасность:** включено шифрование WebSocket соединения
- **Среда разработки:** использует HTTP протокол для локальной работы

**Важные принципы:**
- Все параметры должны точно соответствовать `reverb.php` для корректной работы
- В production настройки могут отличаться (HTTPS, внешний хост)

### Environment переменные

Настройки окружения для WebSocket сервера (Laravel Reverb):

```env
REVERB_APP_ID=powerapp              # Идентификатор приложения
REVERB_APP_KEY=powerapp-key         # Публичный ключ для подключения клиентов
REVERB_APP_SECRET=powerapp-secret   # Секретный ключ для авторизации каналов
REVERB_HOST=127.0.0.1               # Хост WebSocket сервера
REVERB_PORT=6007                    # Порт WebSocket сервера (default)
REVERB_SERVER_HOST=0.0.0.0          # Хост для запуска Reverb сервера
REVERB_SERVER_PORT=8080             # Порт для запуска Reverb сервера
```

### Flutter конфигурация

Flutter приложение получает конфигурацию WebSocket через API endpoint `/api/v3/mobile-variables`. Конфигурация хранится в файле `storage/app/public/mobile-variables/variables_powerapp.json` и передается в зашифрованном виде.

**Параметры конфигурации WebSocket:**

| Поле | Тип | Default | Описание |
|------|-----|---------|----------|
| prod_url | string | - | URL API сервера |
| prod_app_key | string | - | Ключ для WebSocket подключения (должен совпадать с REVERB_APP_KEY) |
| cluster | string | - | Кластер для совместимости с Pusher протоколом |
| port | integer | 6008 | Порт WebSocket сервера |

Инициализация WebSocket клиента происходит в `pusher_service.dart` с использованием параметров из конфигурации.

## Каналы и события

### Структура каналов

| Канал | Тип | Назначение |
|-------|-----|------------|
| `user.{userId}` | Private | Персональные события пользователя |
| `sbp.{clientId}` | Private | События СБП (Система быстрых платежей) для конкретного клиента |

### События аренды

| Event | Канал | Описание |
|-------|-------|----------|
| `App\Events\RentStart` | `user.{userId}` | Успешное начало аренды |
| `App\Events\RentEnd` | `user.{userId}` | Завершение аренды |
| `App\Events\RentFails` | `user.{userId}` | Ошибка при аренде |

**Структуры событий:**

**RentStart содержит:**
- `image` - путь к изображению powerbank
- `title` - локализованный заголовок успеха
- `text` - локализованный текст сообщения
- `coordinates` - координаты станции (lat, lon)
- `station_id` - ID станции

**RentEnd содержит:**
- `title`, `text` - локализованные сообщения
- `station_start_id`, `station_id` - ID станций начала/конца
- `start_coordinates`, `coordinates` - координаты станций
- `rent_id` - ID аренды
- `time_start`, `time_end` - timestamp начала/конца
- `price_format` - отформатированная цена
- `debt` - долг клиента
- `point_start_name`, `point_end_name` - названия точек
- `receipt_url` - ссылка на чек

**RentFails содержит:**
- `title` - локализованный заголовок ошибки
- `text` - сообщение об ошибке


### События СБП

| Event | Канал | Описание |
|-------|-------|----------|
| `App\Events\SbpAddEvent` | `sbp.{clientId}` | Статус обработки СБП платежа |

**SbpAddEvent содержит:**
- `client_id` - ID клиента 
- `status` - статус СБП операции (`success`, `pending`, `failure`)

**Триггеры события:**
- При успешном создании СБП платежа и ожидании подключения - статус `pending`
- При успешном подключении СБП способа оплаты - статус `success`  
- При отклонении пользователем СБП подключения (код ошибки 6001) - статус `failure`
- При получении callback от CloudPayments о статусе авторизации - статус `pending`

### Системные события

Технические работы, экстренные уведомления, обновления статуса станций реализуются по мере необходимости.

## Клиентская интеграция

### Flutter WebSocket Service

**Архитектура клиентского сервиса:**

Flutter приложение использует Riverpod provider для управления WebSocket соединением:

**Основные компоненты:**
- `WebSocketService` - главный сервис для управления соединением
- `PusherChannelsFlutter` - singleton для подключения к Pusher
- `Channel` - объект канала для получения событий
- `AuthService` - провайдер аутентификации для JWT токенов

**Жизненный цикл соединения:**
1. **Инициализация** - настройка Pusher с параметрами подключения
2. **Подключение** - установка WebSocket соединения 
3. **Авторизация** - подписка на private канал с JWT токеном
4. **Привязка событий** - регистрация обработчиков для событий аренды
5. **Автоматическое переподключение** - восстановление при обрыве связи

**Безопасность:**
- Обязательная авторизация для private каналов
- JWT токен в заголовках для проверки прав доступа
- TLS шифрование всего трафика

### Обработка событий

**Паттерн обработки WebSocket событий:**

Flutter приложение использует типизированные обработчики для каждого типа события:

**RentStart (Начало аренды):**
- Десериализация JSON в `RentStartEvent` объект
- Обновление состояния аренды через Riverpod providers
- Показ push-уведомления пользователю
- Обновление UI с информацией об активной аренде

**RentEnd (Завершение аренды):**
- Обработка события завершения аренды
- Обновление статуса аренды на "завершена"
- Показ итогового экрана с суммой и временем
- Очистка состояния активной аренды

**RentFails (Ошибка аренды):**
- Обработка ошибок при попытке взять пауэрбанк
- Показ диалога с описанием проблемы
- Логирование ошибки для диагностики
- Возврат пользователя к экрану карты станций

**Принципы обработки:**
- Валидация входящих данных перед использованием
- Типобезопасная десериализация через `fromJson`
- Централизованное управление состоянием через Riverpod
- Логирование всех событий для отладки

## Безопасность и авторизация

### Авторизация каналов

**Private каналы** (`/api/routes/channels.php`):
```php
Broadcast::channel('user.{userId}', function ($user, $userId) {
    return (int) $user->id === (int) $userId;
});

Broadcast::channel('sbp.{clientId}', function (Client $user, int|string $clientId) {
    return $user->id === (int) $clientId;
});
```

**Авторизация СБП канала:**
- Канал `sbp.{clientId}` доступен только клиенту с соответствующим ID
- Поддерживает как целочисленные, так и строковые идентификаторы клиентов
- Приведение к integer для сравнения обеспечивает безопасность

**Присоединение к каналу:**

Подписка на private канал требует JWT авторизации:
- Канал формируется как `user.{userId}` для персональных событий
- JWT токен передается в заголовке `Authorization` для проверки прав доступа
- Backend валидирует токен и user_id перед разрешением подписки

### Валидация токенов

**Процесс авторизации**:
1. Клиент отправляет JWT токен при подключении
2. API проверяет действительность токена
3. Авторизация канала на основе user_id из токена
4. Подключение к персональному каналу

### Шифрование данных

**Transmission Security**:
- WSS (WebSocket Secure) для зашифрованного соединения
- TLS 1.3 для защиты данных в transit
- Валидация origin для предотвращения CSRF

## Мониторинг и отладка

### Диагностика подключений

**Connection статусы**:
- `CONNECTING` - Установка соединения
- `CONNECTED` - Активное соединение
- `DISCONNECTED` - Отключено
- `RECONNECTING` - Попытка переподключения

**Flutter диагностика:**

**Отслеживание состояния подключения:**
- Мониторинг изменений состояния WebSocket (`CONNECTING`, `CONNECTED`, `DISCONNECTED`, `RECONNECTING`)
- Автоматическая обработка разрывов соединения
- Логирование всех изменений состояния для отладки

**Обработка ошибок:**
- Перехват и логирование WebSocket ошибок
- Отправка критических ошибок в Sentry для мониторинга
- Структурированное логирование для удобства отладки

### Логирование событий

**Система логирования WebSocket событий:**

**Backend (Laravel):**
- Логирование в `broadcastWith()` методах Event классов
- Запись имени события, user_id и данных
- Использование стандартной Laravel Log фасада
- Интеграция с системой мониторинга Sentry

**Frontend (Flutter):**
- Структурированное логирование через logger пакет
- Разделение на info/debug уровни
- Отправка метрик в аналитику для отслеживания
- Логирование всех WebSocket событий для диагностики

### Обработка ошибок

**Типы ошибок**:
- `CONNECTION_FAILED` - Не удалось подключиться
- `AUTHORIZATION_FAILED` - Ошибка авторизации канала
- `CHANNEL_SUBSCRIPTION_FAILED` - Не удалось подписаться на канал
- `EVENT_PARSING_ERROR` - Ошибка парсинга события

**Стратегии восстановления:**

**CONNECTION_FAILED** - Отложенное переподключение:
- Попытка переподключения через 5 секунд
- Экспоненциальное увеличение интервала при повторных ошибках
- Максимум 10 попыток переподключения

**AUTHORIZATION_FAILED** - Обновление токенов:
- Обновление JWT токена через refresh endpoint
- Повторная авторизация каналов с новым токеном
- Logout при невозможности обновить токен

**CHANNEL_SUBSCRIPTION_FAILED** - Переподписка:
- Повторная подписка на все активные каналы
- Восстановление обработчиков событий
- Уведомление пользователя при критических ошибках

### Метрики и производительность

**Система мониторинга производительности:**

**Собираемые метрики:**
- Время установки WebSocket соединения
- Количество активных соединений по времени  
- Частота получения событий по типам
- Процент успешной доставки событий
- Время отклика на WebSocket события

**Интеграция с аналитикой:**
- Автоматическая отправка метрик в Analytics систему
- Трекинг пользовательского опыта с WebSocket
- Мониторинг и алерты при критических ошибках
- Уведомления при деградации производительности
