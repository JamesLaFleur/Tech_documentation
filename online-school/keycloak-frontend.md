## Взаимодействие KeyCloak с Frontend частью


Для упрощения процесса авторизации в данном проекте используется модуль @nuxtjs/auth-next. Данный модуль NuxtJS устанавливается через менеджер пакетов npm:

```
npm install @nuxtjs/auth-next

```

Данный пакет (после его установки) настраивается в файле nuxt.config.js

## Настройка пакета @nuxtjs/auth-next

### Redirect

1. В рамках настройки определяются пути, на которые пользователь будет перенаправлен в определенных случаях:
			
login: '/auth/login',
logout: '/auth/logout',
home: '/',
callback: '/auth/login'

'/auth/login' - по данному пути в файле index.vue, происходит проверка, если пользователь не был авторизован, то происходит вызов метода loginWith('keycloak') из объекта auth, реализуя авторизацию с помощью KeyCloak.

'/auth/logout' - реализован функционал логаута. При выходе из профиля, происходит запрос на logout из KeyCloak
После успешного получения ответа, данные о пользователе удаляются из хранилища.

### Plugins

Плагин подключает необходимые компоненты для процесса авторизации и позволяют глобально использовать переменную пакета(auth-next) внутри проекта.

### Strategies

Стратегии описывают поддерживаемые методы авторизации.

В данном случае доступны две стратегии:

1. Локальная стратегия(local):

В рамках данной стратегии frontend запрашивает данные у backend и сохраняет полученные данные.
В данный момент локальная схема не используется.

2. Стратегия KeyCloak(OpenIDConnect):

По умолчанию OpenIDConnect предоставляет весь необходимый функционал в коробке, для ее использования, необходимо подключить данную схему:

```
scheme: '~/helpers/auth/customOpenIDConnectScheme.ts'

```

В данном случае происходит расширение базовой версии OpenIdConnect и дополнение ее дополнительным функционалом. Для этого был реализован класс CustomScheme.

### CustomScheme

Данная схема расширяет OpenIDConnectScheme.

Базово, с помощью данной схемы, из токена происходит получение пользовательской информации и полученная информация присваивается переменной auth.

В данную схему были внесены следующие изменения:

- После успешного получения информации о пользователе, необходимо также получить данные с бэкенда о данном пользователе и сохранить их в хранилище, так как в KeyCloak хранится только пользовательская информация, необходимая для авторизации, такая как электронная почта, номер телефона и информация о ролях, остальная информация хранится в базе данных проекта.

- После того, как пользователь вышел из аккаунта, сохраненные данные удаляются из хранилища.

### Компоненты стратегии KeyCloak

1. Схема (CustomScheme)

2. redirectUri: process.env.NUXT_ENV_HOME_URL - страница, на которую будет возвращен пользователь после авторизации

3. logoutRedirectUri: process.env.NUXT_ENV_HOME_URL - страница, на которую будет возвращен пользователь после выхода из профиля

4. сlientID - клиент внутри реалма, лежащий в KeyCloak.

5. Endpoints

| URL | Описание |
| --- | --- |
| `KEYCLOAK_OIDC_URL + '/auth'` | Авторизация |
| `KEYCLOAK_OIDC_URL + '/token'` | Получение токена | 
| `KEYCLOAK_OIDC_URL + '/userinfo'` | Информация о пользователе | 
| `KEYCLOAK_OIDC_URL + '/logout'` | Выход из аккаунта | 


KEYCLOAK_OIDC_URL = KEYCLOAK_BASE_URL + '/protocol/openid-connect'

KEYCLOAK_BASE_URL = process.env.KEYCLOAK_ENV_URL + 'realms/' + process.env.KEYCLOAK_ENV_REALM


6. Scope

scope: ['openid', 'profile', 'offline_access', 'roles'] - массив строк, указывающий какие данные и права доступа будут запрошены при авторизации

- openid - обязательный scope протокола OpenIDConnect, обеспечивающий возможность запрашивать токены, содержащие информацию о пользователе

- profile - данный параметр запрашивает основные данные профиля

- offline_access - позволяет запрашивать refresh token

- roles - запрашивает информацию о ролях пользователя, обеспечивая предоставления доступов в зависимости от роли пользователя.


### Получение роли из токена

На многих страницах реализован функционал получения ролей не с бэкенда, а из токена. Данный функционал реализован в функции validate. Происходит это из-за того, что проверка на наличие данных происходит перед авторизацией пользователя. На данном этапе данные с бэкенда не могут быть получены, поэтому происходит их получение из токена. Данная проверка реализована для ограничения прав для пользователей с определенными ролями.
