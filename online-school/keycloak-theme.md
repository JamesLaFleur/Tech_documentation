## KeyCloak-theme

Данный модуль необходим для создания темы для каждого приложения. Так как у каждого сайта свой стиль, то необходимо изменить базовый интерфейс, предоставляемый KeyCloak.


### Docker

При добавлении новой темы при сборке, в volume необходимо прокинуть путь к необходимой теме.
BACKEND_API_URL: https://api.dev.k8s.ponimayu.qtim.ru - адрес бэкенда, к которому обращается фронтенд.

Для разработки, необходимо установить зависимости, сбилдить обе темы, а потом поднять docker.

Шаблон развернут с помощью yarn и node 16.20.2 версии.

### Providers

Папка providers содержит файл, отвечающий за настройку провайдеров. Провайдеры позволяют пользователю авторизоваться через соцсети. Данный плагин дополняет функционал KeyCloak.

В docker-compose в volume передан файл провайдеров:
`- ./providers:/opt/keycloak/providers`


### Папка Темы(themes)

В данной папке хранятся готовые темы.

При запуске команды yarn build(yarn build:kz), webpack собирает необходимую тему и отправляет файл с темой в папку themes для дальнейшей обработки и приведения к виду, который требует KeyCloak.

В данный момент используются две темы:

- для платформы РФ(ponimau);
- для платформы РК(empatia)


### Webpack


В данном файле хранятся переменные с наименованием тем, которые будут отображаться в KeyCloak.
Название темы зависит от переданного параметра country. Страна выбирается в зависимости от той темы, которая используется. То есть при смене темы будет меняться переменная.
В зависимости от выбранной страны меняется политика конфиденциальности и логотипы. 
Тема по умолчанию задана 'ponimau'.

Также определятся массив страниц, которые будут отрисовываться в зависимости от действий пользователя, например страница регистрации или сброса пароля.
Для каждой страницы используется свой файл index.ts.
Webpack собирает отдельную тему для каждой страницы.

Происходит проброс шрифтов, картинок и документов(политики конфиденциальности). Это нужно для того, чтобы все эти данные лежали в static на сервере.

```
::: details
При работе с данным проектом в dev режиме, hot reload не будет работать
:::
```

Так как для работы KeyCloak необходимо сначала собрать проект, то в dev режиме нет возможности сразу получать информацию с KeyCloak.
Таким образом, при внесении определенного изменения, необходимо заново собирать проект, выполняя команду:

```
yarn build
```

Или для empatia команду:

```
yarn build:kz
```

### Hook

#### useApp

Здесь происходит получение текущей страны, которую мы используем на платформе. Получение страны происходит из env-файла. Страна формируется в webpack.


### Static

#### Messages

В messages хранятся варианты сообщений для каждого языка, доступного на платформе.
Для корректной работы и отображения текста сообщений, в файлах необходимо прописывать кодировку.

#### Базовый шаблон

Данный шаблон использует Freemarker для динамической генерации HTML-страницы с учетом данных, передаваемых с сервера.
Работа с шаблонизатором настроена с помощью js-скрипта, ссылающегося на view.
Данный шаблон в дальнейшем может расширяться.

Есть блок переменных (`script id = environment`), хранящихся в json и берущиеся с KeyCloak, они общие для всех страниц.


В базовом шаблоне, в подключаемом скрипте в json-файле, хранятся такие поля как:
- permissions
- labels
- validations
- errors
- user
- auth
- totp
- env
- locales
- message
- social

Все эти поля описаны в `@types/environment`.

При добавлении нового текста необходимо будет добавлять его в types/environment.dts.


permissions - в данном блоке описаны действия, доступные пользователю.
Данные действия настраиваются в настройках realm.
После, данные настройки попадают в шаблон, в соответствующие блоки.

label -  c помощью label генерируются сообщения, которые будут выводиться на платформе. 
Сообщения хранятся в `static/messages`. Доступны варианты сообщений для разных языков, доступных на платформе.

Каждый шаблон ссылается на данные параметры для получения необходимой информации.

Используя `<#nested "scripts">` в шаблон с помощью скриптов можно добавлять дополнительный функционал.


#### Theme.properties

В шаблон могут быть добавлены свойства для темы(`theme.properties`). В том числе передача url, указание языков, доступных для выбора на платформе и указание родительской темы, от которой наследуются базовые свойства.


### components

Все компоненты импортированы с основной платформы и переписаны на vue3.


### docs

Названия файлов документов в KeyCloak должны быть написаны только на английском. 


### views 

Во view хранятся наборы шаблонов страниц, на которые будет переходить пользователь в зависимости от полученных от KeyCloak данных.

В каждый шаблон подключается основной шаблон. Добавляется необходимый скрипт, ссылающийся на js-файл(index.vue)

KeyCloak при формировании страницы, вызывает метод, в методе указан тот тип шаблона, который будет использоваться.


В каждом шаблоне есть файл index для freemarker: index.ftl - шаблон FreeMarker.
В данном файле импортируется основной шаблон FreeMarker, который будет использоваться для всех страниц.
Также есть index.ts файл, который создает vue-приложения, которые будут интегрированы в страницу.
Происходит получение переменных окружения, которые потом будут переданы во vue-приложение с помощью provide, для того, чтобы эти данные были доступны для компонентов приложения.
На стороне сервера шаблон index.ftl отвечает за вставку URL-адресов и других данных окружения при рендеринге страницы на сервере. Он также подключает необходимые скрипты.
Скрипт загружает приложение Vue.js на клиенте, используя данные окружения, переданные из шаблона FreeMarker. Это обеспечивает доступность всех необходимых URL-адресов и настроек для динамического поведения страницы регистрации.
Таким образом, эти два файла(index.ts и index.ftl) необходимы для настройки темы KeyCloak с использованием FreeMarker (для серверного рендеринга шаблонов) и Vue.js (для клиентской логики).

Большинство шаблонов представляют собой базовый функционал KeyCloak, стилизованных под тему платформы. Страницы login и register содержат дополнительный функционал, расширяющий базовую логику шаблона.

#### Страница login

Для авторизации используется компонент <auth-form> с полями для ввода email и пароля.
При помощи объекта validations осуществляется проверка корректности введенных данных.
Отправка введенных данных блокируется, если одно из основных полей пустое.
Данные отправляются, используя метод POST.
Поддерживается функционал входа через социальные сети.
На странице предусмотрены ссылки для восстановления пароля и перехода на страницу регистрации, на основе данных переданных из хука useLogin.


### Добавление дополнительных инпутов

При отправке запроса в KeyCloak используется тэг form, принимающий в action ссылку. После чего полученные с помощью input данные отправляются на backend.

Во время регистрации могут возникнуть проблемы при выполнении данного flow.

При выполнении первого этапа регистрации(ввод email и номера телефона), данные отправляются на backend.

При переходе на другую форму, форма перестает быть видимой, поэтому не предоставляется возможным получить доступ к инпутам в определенной форме(например на первом этапе регистрации).

Для решения данной проблемы была добавлена кнопка регистрации на этапе ввода пароля. То есть была добавлена форма с базовыми инпутами для заполнения. При нажатии на эту кнопку вызывается форма(form, который никто не видит) и внутри есть уже заполненные поля, которые отправляются на backend.

Это важно, так как пользователь может регистрироваться с хешем по инвайту. Данный хеш не виден, но его необходимо отправить в KeyCloak
